## 用户输入

```text
$ARGUMENTS
```

在继续之前，**必须**考虑用户输入（如果不为空）。用户可以指定：
- `create` - 从头创建新的 CLAUDE.md
- `update` - 改进现有的 CLAUDE.md
- `audit` - 分析并报告当前 CLAUDE.md 的质量
- 特定路径以创建/更新（例如，`src/api/CLAUDE.md` 用于目录级说明）

## 核心原则

**LLM 是无状态的**：CLAUDE.md 是自动包含在每次对话中的唯一文件。它作为 AI 代理进入代码库的主要入职文档。

### 黄金法则

1. **少即是多**：前沿 LLM 可以遵循约 150-200 条指令。Claude Code 的系统提示已经占用约 50 条。保持您的 CLAUDE.md 专注且简洁。

2. **通用适用性**：只包含与每次会话都相关的信息。特定于任务的说明应放在单独的文件中。

3. **不要把 Claude 当作 Linter 使用**：样式指南会膨胀上下文并降低指令跟随能力。使用确定性工具（prettier、eslint 等）代替。

4. **永远不要自动生成**：CLAUDE.md 是 AI 工具的最重要杠杆点。用心手动精心制作。

## 执行流程

### 1. 项目分析

首先，分析当前项目状态：

1. 检查现有的 CLAUDE.md 文件：
   - 根级别：`./CLAUDE.md` 或 `.claude/CLAUDE.md`
   - 目录级：`**/CLAUDE.md`
   - 全局用户配置：`~/.claude/CLAUDE.md`

2. 识别项目结构：
   - 技术栈（语言、框架）
   - 项目类型（monorepo、单应用、库）
   - 开发工具（包管理器、构建系统、测试运行器）

3. 审查现有文档：
   - README.md
   - CONTRIBUTING.md
   - package.json、pyproject.toml、Cargo.toml 等

### 2. 内容策略（是什么、为什么、怎么做）

围绕三个维度构建 CLAUDE.md：

#### 是什么 - 技术与结构
- 技术栈概述
- 项目组织（对 monorepo 尤为重要）
- 关键目录及其用途

#### 为什么 - 目的与背景
- 项目做什么
- 为什么做出某些架构决策
- 每个主要组件负责什么

#### 怎么做 - 工作流程与约定
- 开发工作流程（bun vs node、pip vs uv 等）
- 测试程序和命令
- 验证和构建方法
- 关键的"陷阱"或非显而易见的 requirements

### 3. 渐进式披露策略

对于较大的项目，建议创建 `agent_docs/` 文件夹：

```
agent_docs/
  |- building_the_project.md
  |- running_tests.md
  |- code_conventions.md
  |- architecture_decisions.md
```

在 CLAUDE.md 中，用类似这样的指令引用这些文件：
```markdown
有关详细的构建说明，请参阅 `agent_docs/building_the_project.md`
```

**重要**：使用 `file:line` 引用而不是代码片段，以避免过时的上下文。

### 4. 质量约束

创建或更新 CLAUDE.md 时：

1. **目标长度**：少于 300 行（最好少于 100 行）
2. **无样式规则**：删除任何 linting/格式化说明
3. **无特定任务说明**：移到单独文件
4. **无代码片段**：使用文件引用代替
5. **无冗余信息**：不要重复 package.json 或 README 中的内容

### 5. 必要章节

结构良好的 CLAUDE.md 应包括：

```markdown
# 项目名称

简短的一行描述。

## 技术栈
- 主要语言和版本
- 关键框架/库
- 数据库/存储（如果有）

## 项目结构
[仅对 monorepo 或复杂结构]
- `apps/` - 应用入口点
- `packages/` - 共享库

## 开发命令
- 安装：`command`
- 测试：`command`
- 构建：`command`

## 关键约定
[仅非显而易见、高影响的约定]
- 约定 1 及简要说明
- 约定 2 及简要说明

## 已知问题 / 陷阱
[经常绊倒开发者的事情]
- 问题 1
- 问题 2
```

### 6. 应避免的反模式

**不要包括：**
- 代码样式指南（使用 linters）
- 关于如何使用 Claude 的文档
- 显而易见模式的长篇解释
- 复制粘贴的代码示例
- 通用最佳实践（"编写干净的代码"）
- 特定任务的说明
- 自动生成的内容
- 冗长的 TODO 列表

### 7. 验证清单

定稿前，验证：

- [ ] 少于 300 行（最好少于 100 行）
- [ ] 每行都适用于所有会话
- [ ] 无样式/格式化规则
- [ ] 无代码片段（使用文件引用）
- [ ] 命令已验证可用
- [ ] 对复杂项目使用了渐进式披露
- [ ] 记录了关键陷阱
- [ ] 与 README.md 无冗余

## 输出格式

### 对于 `create` 或默认：

1. 分析项目
2. 按照上述结构起草 CLAUDE.md
3. 提交审查
4. 获得批准后写入适当位置

### 对于 `update`：

1. 阅读现有 CLAUDE.md
2. 对照最佳实践审查
3. 识别：
   - 要删除的内容（样式规则、代码片段、任务特定内容）
   - 要精简的内容
   - 缺失的必要信息
4. 提交更改供审查
5. 获得批准后应用更改

### 对于 `audit`：

1. 阅读现有 CLAUDE.md
2. 生成报告，包括：
   - 当前行数 vs 目标行数
   - 通用适用内容的百分比
   - 发现的反模式列表
   - 改进建议
3. 不要修改文件，只报告

## AGENTS.md 处理

如果用户请求创建/更新 AGENTS.md：

AGENTS.md 用于定义专门的代理行为。与 CLAUDE.md（用于项目上下文）不同，AGENTS.md 定义：
- 自定义代理角色和能力
- 代理特定说明和约束
- 多代理场景的工作流程定义

应用类似的原则：
- 保持专注和简洁
- 使用渐进式披露
- 引用外部文档而不是嵌入内容

## 注意事项

- 在包含命令之前始终验证其可用
- 如有疑问，保留它——少即是多
- 系统提醒告诉 Claude，CLAUDE.md"可能相关也可能不相关"——噪音越多，越容易被忽略
- Monorepo 最受益于清晰的"是什么/为什么/怎么做"结构
- 特定目录的 CLAUDE.md 文件应该更加专注
