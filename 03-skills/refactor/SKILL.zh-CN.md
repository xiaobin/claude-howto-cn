# 代码重构技能

一种基于 Martin Fowler 的《重构：改善既有代码的设计》（第二版）的系统性代码重构方法。此技能强调以测试为保障的安全增量变更。

> "重构是在不改变代码外部行为的前提下，改善其内部结构的过程。" — Martin Fowler

## 核心原则

1. **行为保持**：外部行为必须保持不变
2. **小步前进**：进行微小的、可测试的变更
3. **测试驱动**：测试是安全网
4. **持续进行**：重构是持续的过程，而非一次性事件
5. **协作完成**：每个阶段都需要用户批准

## 工作流程概述

```
阶段 1：研究与分析
    ↓
阶段 2：测试覆盖率评估
    ↓
阶段 3：代码异味识别
    ↓
阶段 4：重构计划制定
    ↓
阶段 5：增量实现
    ↓
阶段 6：审查与迭代
```

---

## 阶段 1：研究与分析

### 目标
- 理解代码库结构和目的
- 确定重构的范围
- 收集业务需求背景

### 需要向用户澄清的问题
在开始之前，澄清：

1. **范围**：哪些文件/模块/函数需要重构？
2. **目标**：你想解决什么问题？（可读性、性能、可维护性）
3. **约束**：有哪些不应更改的区域？
4. **时间压力**：这是否阻塞了其他工作？
5. **测试状态**：测试存在吗？通过了吗？

### 操作
- [ ] 阅读并理解目标代码
- [ ] 识别依赖和集成
- [ ] 记录当前架构
- [ ] 记录任何现有的技术债务标记（TODO、FIXME）

### 输出
向用户呈现发现：
- 代码结构总结
- 识别的问题区域
- 初步建议
- **请求批准继续**

## 阶段 2：测试覆盖率评估

### 为什么测试很重要
> "没有测试的重构就像没有安全带的驾驶。" — Martin Fowler

测试是**安全重构的关键推动者**。没有测试，你有引入 bug 的风险。

### 评估步骤

1. **检查现有测试**
   ```bash
   # 查找测试文件
   find . -name "*test*" -o -name "*spec*" | head -20
   ```

2. **运行现有测试**
   ```bash
   # JavaScript/TypeScript
   npm test

   # Python
   pytest -v

   # Java
   mvn test
   ```

3. **检查覆盖率（如果有）**
   ```bash
   # JavaScript
   npm run test:coverage

   # Python
   pytest --cov=.
   ```

### 决策点：询问用户

**如果测试存在且通过：**
- 继续阶段 3

**如果测试缺失或不完整：**
呈现选项：
1. 先写测试（推荐）
2. 在重构期间逐步添加测试
3. 在没有测试的情况下继续（有风险 - 需要用户确认）

**如果测试失败：**
- 停止。在重构之前修复失败的测试
- 询问用户：我们应该先修复测试吗？

### 测试编写指南（如需要）

对于每个被重构的函数，确保测试覆盖：
- 快乐路径（正常操作）
- 边界情况（空输入、null、边界值）
- 错误场景（无效输入、异常）

使用"红-绿-重构"循环：
1. 写一个失败的测试（红）
2. 使其通过（绿）
3. 重构

## 阶段 3：代码异味识别

### 什么是代码异味？
代码异味的症状。代码异味不是 bug，而是代码可以改进的指标。

### 需要检查的常见代码异味

参见 [references/code-smells.md](references/code-smells.zh-CN.md) 获取完整目录。

#### 快速参考

| 异味 | 迹象 | 影响 |
|-------|-------|--------|
| **长方法** | 方法超过 30-50 行 | 难以理解、测试、维护 |
| **重复代码** | 多处相同的逻辑 | Bug 修复需要在多处进行 |
| **大类** | 职责过多的类 | 违反单一职责 |
| **特性依恋** | 方法更多使用另一个类的数据 | 封装性差 |
| **原始类型痴迷** | 过度使用原始类型而非对象 | 缺少领域概念 |
| **长参数列表** | 4+ 参数的方法 | 难以正确调用 |
| **数据簇** | 相同数据项一起出现 | 缺少抽象 |
| **Switch 语句** | 复杂的 switch/if-else 链 | 难以扩展 |
| **推测性通用性** | "以防万一"的代码 | 不必要的复杂性 |
| **死代码** | 未使用的代码 | 混淆、维护负担 |

### 分析步骤

1. **自动化分析**（如果有脚本可用）
   ```bash
   python scripts/detect-smells.py <file>
   ```

2. **手动审查**
   - 系统地浏览代码
   - 记录每个异味的位置和严重性
   - 按影响分类（Critical/High/Medium/Low）

3. **优先级排序**
   重点关注以下异味：
   - 阻塞当前开发的
   - 导致 bug 或混淆的
   - 影响最多变更代码路径的

### 输出：异味报告

向用户呈现：
- 识别出的异味列表（含位置）
- 每个的严重性评估
- 推荐的优先级顺序
- **请求批准优先级**

## 阶段 4：重构计划制定

### 选择重构方法

对于每种异味，从目录中选择适当的重构方法。

参见 [references/refactoring-catalog.md](references/refactoring-catalog.zh-CN.md) 获取完整列表。

#### 异味到重构的映射

| 代码异味 | 推荐的重构方法 |
|------------|---------------------------|
| 长方法 | 提取方法，用查询替代临时变量 |
| 重复代码 | 提取方法，提升方法，形成模板方法 |
| 大类 | 提取类，提取子类 |
| 特性依恋 | 移动方法，移动字段 |
| 原始类型痴迷 | 用对象替代原始类型，用类替代类型代码 |
| 长参数列表 | 引入参数对象，保留整个对象 |
| 数据簇 | 提取类，引入参数对象 |
| Switch 语句 | 用多态替代条件 |
| 推测性通用性 | 折叠层次，内联类，移除死代码 |
| 死代码 | 移除死代码 |

### 计划结构

使用 [templates/refactoring-plan.md](templates/refactoring-plan.zh-CN.md) 中的模板。

对于每个重构：
1. **目标**：什么代码将变更
2. **异味**：解决什么问题
3. **重构**：应用什么技术
4. **步骤**：详细的微步骤
5. **风险**：可能出问题的地方
6. **回滚**：如需要如何撤销

### 分阶段方法

**关键**：逐步引入重构，按阶段进行。

**阶段 A：快速见效**（低风险，高价值）
- 重命名变量以提高清晰度
- 提取明显的重复代码
- 移除死代码

**阶段 B：结构性改进**（中等风险）
- 从长函数中提取方法
- 引入参数对象
- 将方法移动到适当的类

**阶段 C：架构变更**（较高风险）
- 用多态替代条件
- 提取类
- 引入设计模式

### 决策点：向用户呈现计划

在实现之前：
- 展示完整的重构计划
- 解释每个阶段及其风险
- 获得每个阶段的明确批准
- **询问**："我应该继续阶段 A 吗？"

## 阶段 5：增量实现

### 黄金法则
> "变更 → 测试 → 变绿？ → 提交 → 下一步"

### 实现节奏

对于每个重构步骤：

1. **预检查**
   - 测试通过（绿色）
   - 代码编译

2. **做一个小变更**
   - 按照目录中的机制进行
   - 保持变更最小

3. **验证**
   - 立即运行测试
   - 检查编译错误

4. **如果测试通过（绿色）**
   - 用描述性消息提交
   - 进入下一步

5. **如果测试失败（红色）**
   - 立即停止
   - 撤销变更
   - 分析哪里出了问题
   - 如不清楚，询问用户

### 提交策略

每次提交应该：
- **原子性**：一个逻辑变更
- **可逆性**：易于回滚
- **描述性**：清晰的提交消息

示例提交消息：
```
refactor: 从 processOrder() 中提取 calculateTotal()
refactor: 将 'x' 重命名为 'customerCount' 以提高清晰度
refactor: 移除未使用的 validateOldFormat() 方法
```

### 进度报告

在每个子阶段之后，向用户报告：
- 做的变更
- 测试仍然通过吗？
- 遇到的问题
- **询问**："继续下一批？"

## 阶段 6：审查与迭代

### 重构后清单

- [ ] 所有测试通过
- [ ] 无新的警告/错误
- [ ] 代码成功编译
- [ ] 行为不变（手动验证）
- [ ] 必要时更新了文档
- [ ] 提交历史干净

### 指标对比

运行重构前后的复杂度分析：
```bash
python scripts/analyze-complexity.py <file>
```

呈现改进：
- 代码行数变更
- 圈复杂度变更
- 可维护性指数变更

### 用户审查

呈现最终结果：
- 所有变更的总结
- 前后代码对比
- 指标改进
- 剩余技术债务
- **询问**："您对这些变更满意吗？"

### 下一步

与用户讨论：
- 还有其他异味要解决吗？
- 安排后续重构？
- 在其他地方应用类似变更？

## 重要指南

### 何时停止并询问

在以下情况下始终暂停并咨询用户：
- 对业务逻辑不确定
- 变更可能影响外部 API
- 测试覆盖率不足
- 需要重大的架构决策
- 风险级别增加
- 遇到意外的复杂性

### 安全规则

1. **永远不要在没有测试的情况下重构**（除非用户明确承认风险）
2. **永远不要做大变更**——分成小步骤
3. **永远不要在每次变更后跳过测试运行**
4. **永远不要在测试失败时继续**——先修复或回滚
5. **永远不要假设**——有疑问就问

### 不要做什么

- 不要在重构时添加功能
- 不要在生产紧急情况下重构
- 不要重构你不理解的代码
- 不要过度工程化——保持简单
- 不要一次重构所有内容

## 快速启动示例

### 场景：带重复的长方法

**之前：**
```javascript
function processOrder(order) {
  // 150 行代码，包括：
  // - 重复的验证逻辑
  // - 内联计算
  // - 混合的职责
}
```

**重构步骤：**

1. **确保** processOrder() 有测试
2. **提取** 验证逻辑为 validateOrder()
3. **测试** - 应该通过
4. **提取** 计算为 calculateOrderTotal()
5. **测试** - 应该通过
6. **提取** 通知为 notifyCustomer()
7. **测试** - 应该通过
8. **审查** - processOrder() 现在协调 3 个清晰的函数

**之后：**
```javascript
function processOrder(order) {
  validateOrder(order);
  const total = calculateOrderTotal(order);
  notifyCustomer(order, total);
  return { order, total };
}
```
