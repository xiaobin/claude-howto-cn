## 用户输入

```text
$ARGUMENTS
```

在继续之前，**必须**考虑用户输入。用户应提供：
- **想法/主题**：博客文章的主要概念或主题
- **资源**：用于研究的 URL、文件或参考资料（可选但建议提供）
- **目标受众**：博客文章的受众（可选）
- **语气/风格**：正式、随意、技术性等（可选）

**重要提示**：如果用户请求更新**现有博客文章**，请跳过第 0-8 步，直接从**第 9 步**开始。先阅读现有的草稿文件，然后继续迭代过程。

## 执行流程

按顺序执行以下步骤。**不要跳过步骤或在未获得用户批准的情况下继续。**

### 第 0 步：创建项目文件夹

1. 使用以下格式生成文件夹名称：`YYYY-MM-DD-short-topic-name`
   - 使用今天的日期
   - 从主题创建一个简短、URL 友好的别名（小写、连字符、最多 5 个单词）

2. 创建文件夹结构：
   ```
   blog-posts/
   └── YYYY-MM-DD-short-topic-name/
       └── resources/
   ```

3. 在继续之前向用户确认文件夹创建。

### 第 1 步：研究与资源收集

1. 在博客文章目录中创建 `resources/` 子文件夹

2. 对于每个提供的资源：
   - **URL**：获取并将关键信息保存为 `resources/` 中的 markdown 文件
   - **文件**：在 `resources/` 中阅读并总结
   - **主题**：使用网络搜索收集最新信息

3. 对于每个资源，在 `resources/` 中创建摘要文件：
   - `resources/source-1-[short-name].md`
   - `resources/source-2-[short-name].md`
   - 以此类推

4. 每个摘要应包括：
   ```markdown
   # 来源：[标题/URL]

   ## 关键要点
   - 要点 1
   - 要点 2

   ## 相关引用/数据
   - 引用或统计数据 1
   - 引用或统计数据 2

   ## 与主题的关系
   简要说明相关性
   ```

5. 向用户展示研究摘要。

### 第 2 步：头脑风暴与澄清

1. 根据想法和研究资源，呈现：
   - 从研究中确定的主要主题
   - 博客文章的潜在角度
   - 应涵盖的关键要点
   - 需要澄清的信息空白

2. 提出澄清性问题：
   - 你希望读者记住什么主要收获？
   - 有没有研究中你想强调的具体要点？
   - 目标长度是多少？（短文：500-800 词，中等：1000-1500，长文：2000+）
   - 有没有你想排除的要点？

3. **等待用户回复后再继续。**

### 第 3 步：提出大纲

1. 创建一个结构化大纲，包括：

   ```markdown
   # 博客文章大纲：[标题]

   ## 元信息
   - **目标受众**：[目标人群]
   - **语气**：[风格]
   - **目标长度**：[词数]
   - **主要收获**：[核心信息]

   ## 建议结构

   ### 钩子/引言
   - 开场钩子创意
   - 背景设置
   - 论文陈述

   ### 第 1 节：[标题]
   - 关键要点 A
   - 关键要点 B
   - 来自 [来源] 的支持证据

   ### 第 2 节：[标题]
   - 关键要点 A
   - 关键要点 B

   [继续所有章节...]

   ### 结论
   - 关键要点总结
   - 行动号召或最终思考

   ## 待引用来源
   - 来源 1
   - 来源 2
   ```

2. 向用户展示大纲并**请求批准或修改。**

### 第 4 步：保存已批准的大纲

1. 用户批准大纲后，将其保存到博客文章文件夹中的 `OUTLINE.md`。

2. 确认大纲已保存。

### 第 5 步：提交大纲（如果在 git 仓库中）

1. 检查当前目录是否是 git 仓库。

2. 如果是：
   - 暂存新文件：博客文章文件夹、资源和 OUTLINE.md
   - 创建提交，提交消息：`docs: Add outline for blog post - [topic-name]`
   - 推送到远程

3. 如果不是 git 仓库，跳过此步骤并通知用户。

### 第 6 步：撰写草稿

1. 根据已批准的大纲撰写完整的博客文章草稿。

2. 严格遵循 OUTLINE.md 中的结构。

3. 包括：
   - 引人入胜的带钩子的引言
   - 清晰的章节标题
   - 来自研究的支持证据和示例
   - 章节之间的流畅过渡
   - 有力的带收获的结论
   - **引用**：所有比较、统计数据、数据点和事实陈述必须引用原始来源

4. 将草稿保存为博客文章文件夹中的 `draft-v0.1.md`。

5. 格式：
   ```markdown
   # [博客文章标题]

   *[可选：副标题或标语]*

   [包含内联引用的完整内容...]

   ---

   ## 参考文献
   - [1] 来源 1 标题 - URL 或引用
   - [2] 来源 2 标题 - URL 或引用
   - [3] 来源 3 标题 - URL 或引用
   ```

6. **引用要求**：
   - 每个数据点、统计数据或比较必须有内联引用
   - 使用编号引用 [1]、[2] 等，或命名引用 [来源名称]
   - 将引用链接到末尾的参考文献部分
   - 示例："研究表明，65% 的开发者更喜欢 TypeScript [1]"
   - 示例："React 在渲染速度上比 Vue 快 20% [React Benchmarks 2024]"

### 第 7 步：提交草稿（如果在 git 仓库中）

1. 检查是否在 git 仓库中。

2. 如果是：
   - 暂存草稿文件
   - 创建提交，提交消息：`docs: Add draft v0.1 for blog post - [topic-name]`
   - 推送到远程

3. 如果不是 git 仓库，跳过并通知用户。

### 第 8 步：提交草稿供审查

1. 向用户展示草稿内容。

2. 请求反馈：
   - 总体印象如何？
   - 哪些章节需要扩展或缩减？
   - 需要调整语气吗？
   - 是否有缺失信息？
   - 具体编辑或重写？

3. **等待用户回复。**

### 第 9 步：迭代或定稿

**如果用户请求更改：**
1. 记录所有请求的修改
2. 返回第 6 步，并进行以下调整：
   - 递增版本号（v0.2、v0.3 等）
   - 整合所有反馈
   - 保存为 `draft-v[X.Y].md`
   - 重复第 7-8 步

**如果用户批准：**
1. 确认最终草稿版本
2. 如果用户请求，可选择重命名为 `final.md`
3. 总结博客文章创建过程：
   - 创建的版本总数
   - 版本之间的主要变化
   - 最终词数
   - 创建的文件

## 版本跟踪

所有草稿都保留增量版本：
- `draft-v0.1.md` - 初始草稿
- `draft-v0.2.md` - 第一轮反馈后
- `draft-v0.3.md` - 第二轮反馈后
- 以此类推

这允许跟踪博客文章的演进并在需要时回滚。

## 输出文件结构

```
blog-posts/
└── YYYY-MM-DD-topic-name/
    ├── resources/
    │   ├── source-1-name.md
    │   ├── source-2-name.md
    │   └── ...
    ├── OUTLINE.md
    ├── draft-v0.1.md
    ├── draft-v0.2.md（如果有迭代）
    └── draft-v0.3.md（如果有更多迭代）
```

## 质量提示

- **钩子**：以问题、令人惊讶的事实或相关场景开头
- **流程**：每个段落应与下一个相连
- **证据**：用研究数据支持论点
- **引用**：始终引用来源：
  - 所有统计数据和数据点（例如，"根据 [来源]，75% 的..."）
  - 产品、服务或方法之间的比较（例如，"X 比 Y 快 2 倍 [来源]"）
  - 关于市场趋势、研究发现或基准的事实陈述
  - 使用内联引用，格式：[来源名称] 或 [作者，年份]
- **声音**：保持一致的语调
- **长度**：遵守目标词数
- **可读性**：使用短段落，适当使用项目符号
- **CTA**：以明确的行动号召或发人深省的问题结尾

## 注意事项

- 在规定的检查点始终等待用户批准
- 保留所有草稿版本以备历史记录
- 当提供 URL 时，使用网络搜索获取最新信息
- 如果资源不足，请向用户索取更多或建议额外研究
- 根据目标受众调整语气（技术性、一般性、商业性等）
