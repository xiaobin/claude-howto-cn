# AI 代码生成的整洁代码规则

这些规则指导代码生成，以产生可维护的、专业质量的代码。

## 有意义的名称
- 使用揭示意图的名称，解释为什么存在
- 避免 disinformation 和无意义的区分（例如 `data`、`info`、`manager`）
- 使用可发音、可搜索的名称
- 类名：名词（例如 `UserAccount`、`PaymentProcessor`）
- 方法名：动词（例如 `calculateTotal`、`sendEmail`）
- 避免心理映射和编码（匈牙利命名法、前缀）

## 函数
- 函数保持小规模（理想 < 20 行）
- 只做一件事 —— 单一职责原则
- 每个函数一个抽象层级
- 限制参数：0-2 个理想，最多 3 个，避免标志参数
- 无副作用 —— 函数应该做其名称所说的
- 将命令（改变状态）与查询（返回信息）分离
- 优先使用异常而非错误代码

## 注释
- 代码应自我解释 —— 尽可能避免注释
- 好的注释：法律信息、警告、TODO、公共 API 文档
- 坏的注释：冗余、误导或解释坏代码
- 绝不注释掉代码 —— 删除它（版本控制保留了历史）
- 如果需要注释，考虑重构代码

## 格式化
- 保持文件小而专注
- 垂直格式化：相关概念放在一起，空白行分隔概念
- 水平格式化：限制行长（80-120 字符）
- 使用一致的缩进和团队风格
- 将相关函数组合在一起

## 对象和数据结构
- 对象：将数据隐藏在抽象后面，通过方法暴露行为
- 数据结构：暴露数据，有最少行为
- 迪米特法则：只与直接朋友交谈，避免 `a.getB().getC().doSomething()`
- 不要盲目通过 getter/setter 暴露内部结构

## 错误处理
- 使用异常，而非返回代码或错误标志
- 代码可能失败时首先写 `try-catch-finally`
- 在异常消息中提供上下文
- 不要返回 `null` —— 返回空集合或使用 Optional/Maybe
- 不要传递 `null` 作为参数

## 类
- 小类：以职责衡量，而非行数
- 单一职责原则：一个变更理由
- 高内聚：类变量被许多方法使用
- 低耦合：类之间依赖最少
- 开闭原则：开放扩展，封闭修改

## 单元测试
- 快速、独立、可重复、自我验证、及时（F.I.R.S.T.）
- 每个测试一个断言（或一个概念）
- 测试代码质量等于生产代码质量
- 测试名称可描述被测试内容
- Arrange-Act-Assert 模式

## 代码质量原则
- **DRY（不要重复自己）**：无重复
- **YAGNI（你不需要它）**：不要为假设的未来构建
- **KISS（保持简单）**：避免不必要的复杂性
- **童子军规则**：让代码比你发现时更干净

## 应避免的代码气味
- 过长函数或类
- 重复代码
- 死代码（未使用的变量、函数、参数）
- 特性依恋（方法对其他类更感兴趣）
- 不当亲密（类之间了解太多）
- 过长参数列表
- 原始执念（过度使用原始类型而非小对象）
- Switch/case 语句（考虑多态）
- 临时字段（仅有时使用的类变量）

## 并发
- 将并发代码与其他代码分离
- 限制同步/锁定数据的作用域
- 使用线程安全集合
- 保持同步部分小
- 了解你的执行模型和原语

## 系统设计
- 将构造与使用分离（依赖注入）
- 复杂对象创建使用工厂、构建器
- 面向接口编程，而非实现
- 优先组合而非继承
- 当它们简化而非炫耀时应用设计模式

## 重构
- 持续重构，而非大批量
- 前后总有通过的测试
- 小步前进：一次一个更改
- 常见重构：提取方法、重命名、移动、内联

## 文档
- 自文档化代码 > 注释 > 外部文档
- 公共 API 需要清晰文档
- 文档中包含示例
- 文档贴近代码（理想情况下在代码中）

---

**核心哲学**：代码阅读次数是编写次数的 10 倍。为可读性和可维护性优化，而非聪明。
