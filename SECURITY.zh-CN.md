# 安全政策

## 概述

Claude How To 项目安全对我们很重要。本文档概述了我们的安全实践，并描述了如何负责任地报告安全漏洞。

## 支持的版本

我们为以下版本提供安全更新：

| 版本 | 状态 | 支持至 |
|---------|--------|---------------|
| Latest（main） | 有效 | 当前 + 6 个月 |
| 1.x 版本 | 有效 | 直至下一主要版本 |

**注意**：作为教育指南项目，我们专注于维护当前最佳实践和文档安全，而非传统版本支持。更新直接应用到 main 分支。

## 安全实践

### 代码安全

1. **依赖管理**
   - 所有 Python 依赖都固定在 `requirements.txt` 中
   - 通过 dependabot 和手动审查定期更新
   - 每次提交通过 Bandit 进行安全扫描
   - pre-commit 钩子用于安全检查

2. **代码质量**
   - 使用 Ruff 进行 linting 发现常见问题
   - 使用 mypy 进行类型检查，防止类型相关漏洞
   - pre-commit 钩子强制执行标准
   - 所有更改在合并前经过审查

3. **访问控制**
   - `main` 分支保护
   - 合并前需要审查
   - 状态检查在合并前必须通过
   - 仓库写入访问受限

### 文档安全

1. **示例中无密钥**
   - 示例中的所有 API 密钥都是占位符
   - 凭据绝不硬编码
   - `.env.example` 文件显示所需变量
   - 关于密钥管理的清晰警告

2. **安全最佳实践**
   - 示例展示安全模式
   - 文档中高亮安全警告
   - 链接到官方安全指南
   - 在相关部分讨论凭据处理

3. **内容审查**
   - 所有文档都经过安全问题审查
   - 贡献指南中包含安全注意事项
   - 验证外部链接和引用

### 依赖安全

1. **扫描**
   - Bandit 扫描所有 Python 代码中的漏洞
   - 通过 GitHub 安全警报进行依赖漏洞检查
   - 定期手动安全审计

2. **更新**
   - 安全补丁及时应用
   - 仔细评估主要版本
   - 变更日志包含安全相关更新

3. **透明度**
   - 安全更新记录在提交中
   - 漏洞披露负责任处理
   - 适时发布公开安全公告

## 报告漏洞

### 我们关注的安全问题

我们感谢以下报告：
- 脚本或示例中的**代码漏洞**
- Python 包中的**依赖漏洞**
- 任何代码示例中的**加密问题**
- 文档中的**认证/授权缺陷**
- 配置示例中的**数据暴露风险**
- **注入漏洞**（SQL、命令等）
- **SSRF/XXE/路径遍历**问题

### 不在范围内的问题

以下超出本项目范围：
- Claude Code本身的漏洞（向 Anthropic 报告）
- 外部服务或库的漏洞（向上游报告）
- 社会工程或用户教育（不适用于本指南）
- 无概念证明的理论漏洞
- 通过官方渠道报告的依赖漏洞

## 如何报告

### 私密报告（首选）

**对于敏感安全问题，请使用 GitHub 的私密漏洞报告：**

1. 访问：https://github.com/luongnv89/claude-howto/security/advisories
2. 点击"Report a vulnerability"
3. 填写漏洞详情
4. 包括：
   - 漏洞的清晰描述
   - 受影响组件（文件、部分、示例）
   - 潜在影响
   - 重现步骤（如适用）
   - 建议修复（如有）

**接下来会发生：**
- 我们将在 48 小时内确认收到
- 我们将调查并评估严重程度
- 我们将与您合作制定修复方案
- 我们将协调披露时间表
- 除非您希望匿名，否则我们将在安全公告中致谢您

### 公开报告

对于非敏感或已公开的问题：

1. **开启 GitHub Issue**，标签为 `security`
2. 包括：
   - 标题：`[SECURITY]` 后跟简要描述
   - 详细描述
   - 受影响文件或部分
   - 潜在影响
   - 建议修复

## 漏洞响应流程

### 评估（24 小时）

1. 我们确认收到报告
2. 使用 [CVSS v3.1](https://www.first.org/cvss/v3.1/specification-document) 评估严重程度
3. 我们确定是否在范围内
4. 我们联系您，提供初步评估

### 开发（1-7 天）

1. 我们开发修复方案
2. 我们审查并测试修复
3. 我们创建安全公告
4. 我们准备发布说明

### 披露（因严重程度而异）

**严重（CVSS 9.0-10.0）**
- 立即发布修复
- 发布公开公告
- 提前 24 小时通知报告者

**高（CVSS 7.0-8.9）**
- 48-72 小时内发布修复
- 提前 5 天通知报告者
- 发布时公开公告

**中（CVSS 4.0-6.9）**
- 在下次定期更新中发布修复
- 发布时公开公告

**低（CVSS 0.1-3.9）**
- 包含在下下次定期更新中
- 发布时公告

### 发布

我们发布的安全公告包括：
- 漏洞描述
- 受影响组件
- 严重程度评估（CVSS 分数）
- 修复版本
- 变通方案（如适用）
- 报告者致谢（经允许）

## 报告者最佳实践

### 报告前

- **验证问题**：您能始终如一地重现吗？
- **搜索现有问题**：已经报告了吗？
- **检查文档**：有安全使用指导吗？
- **测试修复**：您建议的修复有效吗？

### 报告时

- **具体**：提供确切的文件路径和行号
- **包括上下文**：为什么这是安全问题？
- **展示影响**：攻击者能做什么？
- **提供步骤**：我们如何重现？
- **建议修复**：您会如何修复？

### 报告后

- **耐心**：我们资源有限
- **响应迅速**：快速回答后续问题
- **保密**：在修复前不要公开披露
- **尊重协调**：遵循我们的披露时间表

## 安全标头和配置

### 仓库安全

- **分支保护**：Main 分支更改需要 2 次批准
- **状态检查**：所有 CI/CD 检查必须通过
- **CODEOWNERS**：关键文件的指定审查者
- **签名提交**：推荐贡献者使用

### 开发安全

```bash
# 安装 pre-commit 钩子
pre-commit install

# 本地运行安全扫描
bandit -c pyproject.toml -r scripts/
mypy scripts/ --ignore-missing-imports
ruff check scripts/
```

### 依赖安全

```bash
# 检查已知漏洞
pip install safety
safety check

# 或使用 pip-audit
pip install pip-audit
pip-audit
```

## 贡献者安全指南

### 编写示例时

1. **绝不硬编码密钥**
   ```python
   # 错误
   api_key = "sk-1234567890"

   # 正确
   api_key = os.getenv("API_KEY")
   ```

2. **警告安全影响**
   ```markdown
   安全注意：绝不将 `.env` 文件提交到 git。
   立即添加到 `.gitignore`。
   ```

3. **使用安全默认值**
   - 默认启用身份验证
   - 适用处使用 HTTPS
   - 验证和清理输入
   - 使用参数化查询

4. **记录安全注意事项**
   - 解释为什么安全重要
   - 展示安全 vs. 不安全模式
   - 链接到权威来源
   - 突出显示警告

### 审查贡献时

1. **检查暴露的密钥**
   - 扫描常见模式（api_key=、password=）
   - 审查配置文件
   - 检查环境变量

2. **验证安全编码实践**
   - 无硬编码凭据
   - 适当的输入验证
   - 安全的身份验证/授权
   - 安全的文件处理

3. **测试安全影响**
   - 这能被滥用吗？
   - 最坏情况是什么？
   - 有边缘情况吗？
